WSB-PIU Task Manager

Dokumentacja projektowa · Projektowanie Interfejsów Użytkownika · WSB Merito, sem. 4 · prowadzący: mgr inż. Bartłomiej Kizielewicz · autor: Darek

Słownik (dla osób nietechnicznych): frontend — warstwa aplikacji, którą widzi i klika użytkownik; modal — okienko nakładane na ekran (formularz), bez przechodzenia na osobną stronę; user flow — ścieżka kroków, którą przechodzi użytkownik do celu; Hi-Fi — szczegółowy projekt wizualny w docelowych kolorach i czcionkach (Figma); design token — nazwana wartość projektu (np. „kolor tła"), używana spójnie w całej apce; localStorage — pamięć przeglądarki, dane zostają po zamknięciu karty, bez logowania; responsywność — dopasowanie układu do szerokości ekranu; a11y — dostępność dla osób z niepełnosprawnościami.

1 · Problem i persony

Problem projektowy

Narzędzia do zarządzania zadaniami są albo zbyt rozbudowane (Jira, Trello, Asana — krzywa nauki, przytłaczające), albo zbyt prymitywne (karteczki, notatki w telefonie — gubią się, brak kategorii i terminów). Brakuje rozwiązania pośrodku: lekkiego i intuicyjnego, w którym dodajemy zadanie w kilka sekund, a mimo to mamy kategorie, priorytety i terminy.

Persony użytkowników (Etap 1)

Kasia, 22 — studentka informatyki. Wiele projektów zespołowych i kolokwiów naraz, korzysta głównie z telefonu, między zajęciami. Karteczki gubi, notatki to bałagan, Trello za rozbudowane.
Kluczowa potrzeba: szybkość i mobilność — dodać zadanie w 3 sekundy, jedną ręką.
Marek, 28 — specjalista ds. marketingu, projekt po godzinach. Świadomie nie miesza spraw prywatnych z firmowymi narzędziami (prywatność, work-life balance). Potrzebuje jednego miejsca na side-project i sprawy życiowe.
Kluczowa potrzeba: kategorie (prywatne vs projekt), priorytety, terminy, filtrowanie.
Anna, 58 — nauczycielka. Sprawdziany, dyżury, dom. Próbowała aplikacji do zadań, ale były „za napakowane" — za dużo opcji, za małe przyciski.
Kluczowa potrzeba: prostota, czytelność, duże elementy klikalne, jasne komunikaty.

2 · Główny user flow i kluczowe decyzje projektowe

Główny przepływ użytkownika

Otwarcie aplikacji (bez logowania — dane w localStorage) │ ▼ Lista zadań ───────────────────────────────────────────────┐ • grupowanie po kategorii (kolorowe nagłówki) │ • sekcja „Dzisiaj" na górze │ filtr / wyszukiwanie / • wyraźne statusy (do zrobienia ↔ zrobione) │ sortowanie — in-place │ │ (bez przeładowania) ├──► „+ Dodaj zadanie" → MODAL formularza ───────────┘ │ tytuł + termin od razu; priorytet / kategoria / │ opis pod „Więcej opcji" → Zapisz / „Zapisz i nowe" │ ├──► klik w checkbox → wykonane (check + przekreślenie + przyciemnienie) │ ├──► edycja → ten sam MODAL (wypełniony danymi) │ └──► usunięcie → potwierdzenie → toast „Cofnij" (5 s) Ustawienia (/settings): motyw (jasny / ciemny / system) + zarządzanie kategoriami

5 kluczowych decyzji projektowych (z uzasadnieniem)

  1. Dodawanie/edycja w modalu, nie na osobnej stronie. Z journey map: Marek dodaje zadania w kontekście listy, Anna po zapisie wraca do identycznego widoku. Modal zamyka pętlę bez nawigacji. (ADR-0001/0002, plan Etap 5)
  2. Status dwukanałowo, nie samym kolorem. Zrobione = zielony check + przekreślenie + przyciemnienie. Powód: ~8% mężczyzn nie rozróżnia czerwieni/zieleni; Anna czyta status z 1 m. Zgodne z WCAG 1.4.1.
  3. Formularz z progresywnym ujawnianiem. Na starcie tylko tytuł + termin; priorytet/kategoria/opis pod „Więcej opcji". Godzi prostotę (Anna) z kontrolą (Marek) jednym komponentem. Etykiety nad polami, komunikaty błędów po ludzku.
  4. Mobile-first, duże elementy klikalne. Układ najpierw pod telefon (Kasia), sidebar zwija się w wysuwane menu. Przyciski min. 44×44 px, checkbox min. 32×32 px, stałe miejsce zapisu na dole.
  5. Stan na useReducer + Context, dane w localStorage; stan widoku osobno. Auto-zapis po każdej akcji, bez przycisku „Zapisz" i bez logowania. Filtry/wyszukiwanie/sortowanie świadomie poza zapisem (czysta funkcja selectVisibleTasks), by filtrowanie było płynne. (ADR-0003)

Bonus (most projekt → kod): kolory, czcionki i kształty z prototypu Hi-Fi (Figma) zmapowano 1:1 na design tokeny (mechanizm @theme w Tailwind), m.in. tło #f4efe4, tekst #221f19, czcionka Inter. Aplikacja wygląda jak prototyp, a tryb ciemny to podmiana tokenów.

3 · Testy użyteczności (metodologia + wnioski)

Metodologia

Zaprojektowano kompletny zestaw testów (Etap4_Usability_Tests.md): moderowany test typu think-aloud (myślenie na głos) na działającej aplikacji, z 3 uczestnikami wg person (student ~20 / pracujący ~25–35 / osoba mniej techniczna 50+). Podstawa: badania Jakoba Nielsena — 3 użytkowników wykrywa ok. 60–75% problemów użyteczności; liczy się różnorodność profili, nie wielkość próby.

Scenariusz (5 zadań): (Z1) dodaj zadanie z priorytetem i kategorią, (Z2) oznacz wykonane, (Z3) filtruj po kategorii, (Z4) zmień termin, (Z5) włącz tryb ciemny. Mierzone: sukces zadania (1/0), orientacyjny czas, liczba błędów, kwestionariusz SUS (System Usability Scale, 0–100) i cytaty jakościowe.

4 · Implementacja i retrospektywa

Stack technologiczny

  • Build: Vite 8 · Framework: React 19 + TypeScript 6 (tryb strict, 0 użyć any).
  • Styling: Tailwind CSS v4 + shadcn/ui (na Radix — dostępność z definicji); tokeny z Figmy przez @theme.
  • Routing: react-router v7 (/, /settings, *) z dzieleniem kodu. Stan: useReducer + Context.
  • Formularze: react-hook-form + zod (jeden schemat waliduje formularz i localStorage). Daty: date-fns.
  • Testy: Vitest + Testing Library (49) + Playwright/Chromium (13) = 62 testy, wszystkie zielone. Jakość: ESLint, Prettier, Husky. CI/CD: GitHub Actions + Vercel.

Zrealizowane funkcje 12 = 5 wymaganych + 7 rozszerzonych

5 wymaganych: dodawanie · edycja · usuwanie (z „Cofnij") · oznaczanie wykonane · lista zadań.
7 rozszerzonych: filtrowanie + sortowanie · priorytety + terminy · kategorie/tagi · wyszukiwarka (debounce) · tryb ciemny (jasny/ciemny/system, zapamiętywany) · walidacja formularzy (PL) · localStorage (wersjonowanie + odzyskiwanie uszkodzonych danych).

Liczby (stan QA, Etap 6)

  • 62 testy automatyczne zielone (49 jednostkowych/komponentowych + 13 e2e).
  • Główny bundle ~117 kB po gzip (z 208 kB przed podziałem); formularz i ustawienia ładowane dopiero, gdy potrzebne (lazy).
  • Responsywność 375 / 768 / 1280 px — brak poziomego przewijania; wykryto i naprawiono błąd paska narzędzi na 768 px.
  • Dostępność: pełna obsługa klawiaturą, status nie tylko kolorem, etykiety + ARIA, lang="pl" (audyt kodu).

Czego się nauczyłem

  • Testowanie na prototypie przed implementacją jest tańsze niż przerabianie działającego kodu.
  • Decyzje warto zapisywać od razu (ADR + CHANGELOG) — po tygodniach łatwo zapomnieć „dlaczego tak".
  • Dostępność to nie dodatek na koniec — projektowanie od początku było tańsze niż łatanie.
  • Progresywne ujawnianie pogodziło sprzeczne potrzeby person jednym komponentem.
  • TypeScript strict wymusił przemyślenie modelu danych przed UI — mniej błędów później.
